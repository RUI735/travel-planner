旅游规划产品 PRD（产品需求文档）| 产品暂定名 | 旅游规划（Travel Planner） |---1. 背景与问题陈述1.1 背景自由行用户在行前需要整合：目的地信息、景点开放时间、交通衔接、预算与住宿位置。现有方式多为碎片化搜索、自行表格排程，耗时且易遗漏闭馆、预约、通勤过远等问题。1.2 要解决的问题需求表达成本高：自然语言描述易被误解，反复修改成本高。路线可执行性弱：缺少对开放时间窗、体力/人群约束、交通耗时的系统化整合。住宿与行程脱节：常见攻略「景点一堆、住宿另说」，未按「每日通勤」优化。1.3 产品目标（可衡量方向）用户通过 结构化输入（选择题为主，基础槽位可跳过） 在 10 分钟内（可缩短为极简路径）获得 至少 1 套 可执行的每日时间表（含 逐段 Leg 距离与用时、交通与关键注意事项）。（MVP1） 结合 每日天气预报 生成 多套可对比方案，阴天/雨天优先室内与短户外动线，并 可解释。分阶段交付：先验证「日程可信 + Leg 可执行」，再叠加天气差异化多方案与地图，最后叠加行前刷新与预订闭环。---2. 术语表术语	定义问卷 / 槽位	用户选择题与明确输入项，映射为结构化字段（JSON）。行程方案	一套完整的按天排程 + 交通摘要 +（可选）住宿与预算汇总。约束	开放时间、不爬山、9 点后出发等影响排序与可行性的规则。启发式路线	在开放时间与交通偏好下近似最优的游览顺序，不承诺数学全局最优。MVP0 / MVP1 / MVP2	见第 5 节范围分层。基础槽位（可跳过）	目的地、人数、出发/返程日期、预算等结构化字段；未填时走 默认推断 或 生成前补全校验（见 7.1.4）。当日起点 / 出发地	用户指定的 当日第一段行程的起点（如酒店、车站、机场、上一日住宿）；用于计算「起点 → 首景点」及各段衔接。行程段（Leg）	相邻两个节点之间的单段路径：起点→景点1、景点1→景点2…；每段输出 距离、用时、推荐交通方式、费用区间。天气驱动排程	按 每日预报天气类型 调整室内外/海边/观景类 POI 权重，并写入可解释说明。---3. 产品定位与核心价值3.1 一句话定位用户通过完成一系列 选择题与明确输入，由系统生成 完整、可执行、可解释 的旅游攻略；在后续版本中逐步加入 地图路线可视化、多方案对比 与 住宿区域/预订辅助。3.2 核心价值（对外叙事顺序）可执行日程优先：按天时间段、闭馆/预约提醒、逐段交通（从当日起点到各景点）与缓冲时间。路线合理可解释：结合地理与交通偏好做顺序优化；结合每日天气调整室内外与海边/户外取景类推荐，并说明「因阴天优先室内」等理由。输入低门槛、可跳过：目的地、人数、出发/返程、预算等 均可填写或暂时跳过，未填项由默认策略推断或生成前引导补全（见 7.1.4）。住宿与通勤匹配（分阶段）：从「区域 + 通勤摘要」升级到「具名备选 + 预订跳转」。可调整：手动改顺序/替换景点后，系统重算交通与摘要（分阶段）。3.3 非目标（当前版本明确不做或弱化）不承诺「全网最低价酒店」「实时房态 100% 准确」（依赖第三方与商务条款）。不在未获授权情况下抓取受版权保护的攻略长文。不替代官方预约与购票渠道的法律义务说明。---必须交付：基础槽位（7.1.4）：目的地、人数、出发日期、返程日期、预算等 均可填或跳过问卷 Step2～Step4 + 旅行偏好 + 交通偏好；住宿模块简化为：预算档位下的 「推荐住宿区域 + 选型建议 + 与当日核心动线的通勤摘要」（不强制输出真实酒店名称，或仅作「示例」并免责声明）。景点筛选 + 按天排程 + 逐段 Leg 交通估算（含 当日起点 → 首景点、景点间；步行 / 打车模型；公共交通为 区间估算 + 换乘提示模板）。结构化输出：行程总览、每日时间表、按 Leg 列出的距离/用时/方式、交通汇总、景点信息汇总、实用提示。可解释块：优化目标、生效约束、关键权衡说明（含闭馆调整）；若已接入天气：说明当日天气对选点/顺序的影响（见 7.3.5）。天气数据：无 API 时输出「天气待接入」且不承诺天气驱动；有 API 时规则见 7.3.5。可选增强：多方案生成（增强）：在原有风格权重（经典 / 省钱 / 深度等）基础上，引入 按日天气预报 的 智能差异化：例如 阴天 / 多云 降低海边、强依赖「通透远景」的户外观景台权重， 提升室内展馆、博物馆、商业室内、剧院等 权重；雨天 进一步强化室内与短室外动线；晴好 恢复海滨、城市观景、公园徒步等权重。每套方案须在 `explainability` 中标注 天气依据日期与数据源更新时间。多方案数量：默认 2～3 套；至少 1 套应体现与天气相关的 差异化动线（若多日天气不同，可按天分别说明）。天气增强：临近出行前的 预报刷新、极端天气（暴雨、大风、高温）预警与行程 自动重建议（可选开关）。闭馆临时公告、人流（依赖数据源与 SLA）。手动编辑：增删改景点、拖拽排序后 局部/全量重算 交通与时间轴；修改当日起点 后重算当日全部 Leg。住宿：具名备选、价格波动提示、OTA 深链；与合作伙伴数据接口（若有）。---6. 用户旅程与关键流程6.1 主路径（Happy Path）```text进入产品 → 填写基础槽位（均可跳过）+ 偏好问卷（可极简）→ 提交  → 系统校验「最小可生成条件」（7.1.4）与互斥项 → 拉取/匹配景点基础数据  →（MVP1+）拉取行程日对应目的地天气预报 → 天气规则参与筛选/打分  → 约束满足 + 路线启发式优化 → 生成 1～3 套方案（含天气差异化说明）  → 展示结构化攻略（含每段 Leg 距离/用时）+（MVP1）地图 → 保存/分享```6.2 异常与降级路径（必须定义）场景	系统行为目的地无足够 POI 数据或缺经纬度	提示缺口；用「区域级推荐」补足；不强行优化顺序。用户约束过严导致无可行解	明确列出冲突约束；给出「放宽建议」（如减少每日景点数、取消某标签）。景点临时闭馆/无开放时间	标注「信息未验证」；建议用户二次确认官方渠道。交通 API 失败	降级为距离 + 时间经验区间 + 打车估算。天气 API 失败	不应用天气驱动权重；方案说明中标注「天气数据暂不可用」；多方案退化为非天气差异化（仅风格权重）。用户跳过出发/返程导致行程日无法确定	使用默认推断（7.1.4）；若仍无法确定，拦截并引导补全日期或选择「默认 2 日游/3 日游」模板。生成超时	异步任务 + 邮件/站内通知（若有账号）；或同步返回简化版（仅天数骨架）。---7. 功能需求详述7.1 需求输入模块（问卷）7.1.1 设计原则结构化优先：基础出行信息与偏好均以 选择题 / 明确控件 为主，减少自然语言依赖。基础槽位可跳过：目的地、人数、出发/返程日期、预算等 每一项均可填写或点击「暂不填写 / 跳过」；未填项进入 推断与补全策略（见 7.1.4），不得因「想先看看」而强迫用户编造信息。极简模式：允许用户在仅满足 最小可生成条件 的前提下直接生成；偏好类 Step 未填时使用 产品默认画像（如休闲舒适 + 经济预算 + 综合偏好），并在输出中标注「以下按默认偏好生成，可返回问卷细化」。人群：主人群单选 + 次要标签多选。旅行风格 / 特殊限制 / 交通 / 住宿：规则同下；未填时走默认。旅行偏好：多选；若用户跳过本步，则使用「综合均衡」默认标签集。住宿：MVP0 以区域与通勤摘要为主；类型与核心需求进入打分（见 7.4）。7.1.2 基础槽位（均可单独跳过）以下字段在 UI 上 逐项提供「跳过」，写入数据模型时为 `null` 或 `skipped: true`：字段	控件形态	跳过时的含义（概要，详见 7.1.4）目的地	热门城市枚举 + 其他手动输入	无法生成真实 POI 动线 → 触发最小条件拦截，须补目的地或选择「灵感城市」（产品可配置快捷入口）出行人数	数字步进器	默认 2 人（可配置）；影响人均预算换算与部分亲子规则出发日期	日期	默认 「下一个周末首日」 或 「今日起第 T 天」（可配置）返程日期	日期	默认 出发日 + 默认行程天数 −1」；若出发也跳过则配合默认天数预算	档位 + 自定义金额	默认 经济型；自定义跳过则不展示精确预算线，仅给参考区间说明：原「出行天数」可与 出发 + 返程 互算：若用户填写起止日期则 天数由系统计算；若跳过日期但填写天数，则 仅生成相对 Day1…DayN（展示时可不绑具体公历直至用户补日期）。若起止与天数均跳过，则使用 默认 2 天 行程骨架。当日起点 / 出发地（强烈建议填写，可跳过）定义：用户指定的 当日第一段 Leg 的起点，如「酒店」「民宿地址」「高铁到达站」「机场」「市中心某路口」。控件：地址搜索 / 地图选点 / 快捷标签（酒店 / 车站 / 机场）；可跳过。跳过：MVP0 使用 「推荐住宿区域几何中心」或目的地市中心/主交通枢纽」 作为默认起点，并在输出中醒目标注「未指定出发地，以下为估算起点」。7.1.3 偏好与细化问卷（Step A～F，均可跳过或部分填写）Step A — 人群与风格主人群单选；次要标签多选；旅行风格单选；特殊限制多选。跳过：默认主人群「朋友结伴」、风格「休闲舒适」、限制「无」（可配置）。Step B — 旅行偏好美食 / 人文历史 / 自然 / 小众 / 购物 / 夜景 / 文化演出等，多选。跳过：默认「综合均衡」标签集。Step C — 交通偏好省时 / 省钱 / 便捷公交 / 自驾。跳过：默认「优先便捷（公共交通为主）」。Step D — 住宿偏好住宿类型 + 核心需求多选。跳过：默认勾选「离景点近」「交通便利」+ 经济型酒店倾向。Step E — 出行天数（若未由日期推导）1 / 2 / 3 / 4～5 / 6～7 / 7+。若已由出发返程推导则本步自动隐藏或只读展示。Step F — 额外需求（可选）结构化模板 + 短自由文本；内容安全与长度限制。7.1.4 未填写处理与「最小可生成条件」硬门槛（不可跳过）：至少具备 有效目的地（用户选择或「灵感模式」替用户选一个城并二次确认）。无目的地则 禁止调用 景点与路线引擎，仅可展示营销页或城市列表。软门槛（可推断）：人数、日期、预算、当日起点、偏好类均可推断；推断结果写入 `inferred_fields[]` 供前端展示「可一键采纳或修改」。日期一致性：若同时填写出发、返程，则 以日期计算天数 为准覆盖 Step E；若冲突（返程早于出发）则拦截。出发日期规则：若填写且早于业务配置的最早可出行日，则提示调整（与原先一致）。7.1.5 校验规则（示例）「不乘坐公共交通」与「优先便捷（公交为主）」互斥 → 提交前拦截并解释。「少走路」与「特种兵」同时选 → 警告并要求确认优先级（默认 特殊限制 > 旅行风格）。基础槽位全跳过且未选灵感城市 → 仅允许进入问卷预览，「生成行程」按钮不可用并提示补 目的地。---7.2 景点信息获取与维护7.2.1 字段（每个景点）基础：名称、地址、经纬度、类别标签、等级/热度（若有）。空间与天气敏感度标签（用于天气驱动排程）：`indoor`（室内为主）、`outdoor`（户外步行多）、`coastal`（海滨沙滩）、`scenic_viewpoint`（强依赖通透视野与光线）、`half_outdoor`（半室外步行街等）。标签可由运营规则 + 模型辅助标注，需支持人工纠错。运营：开放时间规则（含周一闭馆、季节性）、门票参考价、是否需预约、预约渠道说明（链接占位）。体验：建议游玩时长、人群适宜度、避坑提示。合规与安全：禁拍区、证件要求等 引用官方说明 +「以现场为准」。7.2.2 数据 SLA标注数据来源与 最后更新时间；核心字段缺失时不参与自动优化或降级为「待定」。---7.3 智能路线规划7.3.1 景点筛选输入：天数、预算、人群、偏好、特殊限制；（MVP1+）每日天气类型 与 POI 天气敏感度标签。输出：候选集 + 剔除原因日志（供调试与可解释性，可对高级用户折叠展示）。规则示例：「不爬山」过滤高爬升/山地类标签；「不早起」约束首日首个活动开始 ≥9:00；天气规则见 7.3.5。7.3.2 顺序与交通距离：直线距离 + 若有路径 API 则实际路网距离；无则模型估算。交通方式展示（MVP0）步行：≤X km 给出分钟数，费用 0。打车：估算费用区间（起步价 + 里程规则可配置）。公共交通：时间区间 + 费用区间 + 换乘条数上限提示，标明「估算，以地图 App 为准」。优化：在开放时间窗与每日强度上限内做启发式排序（如插入启发式 / 局部 2-opt）；目标函数随「省时/省钱」权重切换。文案：输出中禁止「全局最优」；使用「在当前约束下的推荐顺序」。7.3.3 每日排布与行程段（Leg）输出规范每日景点数随旅行风格：休闲 2～3；特种兵 4～5（可配置表）。景点间插入交通缓冲与简餐/休息块。行程段（Leg）强制输出：对每一天，按时间顺序列出 相邻节点之间的所有 Leg，不得只写「景点 A」而省略如何到达。节点包括：当日起点（用户指定或默认）→ 景点 1 → 景点 2 → … →（可选）餐饮点 → 当日结束点（返回住宿或开放式结束）。每个 Leg 必须包含字段：`from`（名称+类型：起点/景点/餐厅）、`to`、`distance_km`（或米+单位）、`duration_min`（或区间）、`recommended_mode`（步行/公交/打车/自驾）、`cost_estimate_cny`（区间，步行为 0）、`notes`（如换乘简示）。距离与用时优先来自 路径规划 API；失败时 直线距离 + 经验系数 估算，并标注「估算」。与住宿衔接：若当日含「返回住宿」Leg，终点为住宿区域中心或用户指定酒店；与 7.4 住宿摘要一致。末段：当日最后一个游览节点 → 住宿/当日起点，同样输出完整 Leg 字段。7.3.4 餐饮推荐（MVP0）午/晚各 0～1 个推荐位；标注菜系、人均区间、与当前位置距离层级（近/中/远）；不承诺排队与订座。若餐饮点参与当日动线，必须插入在相邻 Leg 序列中（上一节点→餐厅、餐厅→下一节点各为独立 Leg）。7.3.5 天气驱动选点与多方案差异化（MVP1 核心）数据源：按 目的地 + 每日 `date` 拉取天气预报（至少：天气现象、降水概率、气温、风力）；展示 预报发布时间与数据源。天气分类（示例枚举）：`clear`（晴）、`partly_cloudy`（多云）、`overcast`（阴）、`rain`、`snow`、`windy`、`fog`；可由供应商映射归一。权重策略（可配置表）：阴天 / 多云：显著 降低 `coastal`、`scenic_viewpoint` 及强依赖拍照光线的纯户外线；提升 `indoor`、博物馆、商场室内、室内演艺等；理由模板示例：「当日偏阴，海边/远景观感一般，已优先安排室内展馆」。雨天：在阴天策略上进一步强化室内与短室外（连廊、地铁连通商业体）；户外徒步类 降权或剔除（除非用户偏好仅自然且确认冒雨）。晴好：恢复海滨、观景台、城市公园等权重；可适度安排日落/夜景时段。大风 / 极端气温：缆车、海岛船班、长时间无遮蔽户外等 按安全规则降级或提示。与多方案模块联动：多方案不仅按「经典/省钱/深度」权重区分，还须至少体现 天气适配差异（例如方案 A 偏「晴好动线」、方案 B 偏「阴雨友好动线」）；若多日天气不一致，按天分别应用规则并在 `explainability.weather_by_day[]` 中逐日说明。用户强偏好冲突：若用户勾选「自然景观优先」且当日为阴天，系统 不强行删除 全部户外，而是 保留 1 个代表性户外 + 增加室内平衡，并在可解释块中说明权衡。无天气数据：不应用本策略，见 6.2 降级表。---7.4 智能住宿推荐（分阶段）MVP0输出 住宿区域（商圈/片区名称）+ 理由（距核心景点、枢纽关系）+ 与每日动线通勤摘要。给出 类型与价格带 建议；酒店名称仅「示例」或省略。免责声明：价格与房态以 OTA/酒店官方为准。MVP1～2每区域 2～3 个具名备选；外链搜索深链或 API；价格波动与房源紧张提示（有数据源时）。---7.5 输出结构（固定 Schema，便于渲染与 Coze）7.5.1 方案级`plan_id`、`plan_title`、`plan_style`（经典/省钱/天气友好等）`optimization_goal`、`active_constraints[]`、`tradeoff_notes`（可解释）`weather_snapshot`（可选）：预报时间、供应商、`daily_forecast[]`（与方案生成所用一致）`explainability.weather_by_day[]`（MVP1+）：`date`、`condition`、`impact_summary`（如「阴天，已减少海滨户外」）`budget_summary`（门票/交通/餐饮/住宿均为 参考区间）`risk_disclaimers`（数据时效、预约责任；天气预报仅供参考）7.5.2 日级 `days[]``date`、`theme`（可选）`day_start`：当日起点 POI/地址引用、`source`（`user_input` / `default_zone_center`）`segments[]`：时间段起止、类型（景点/交通/餐饮/休息）、关联 POI id、说明、预约提醒`legs[]`（与 segments 并列或嵌套，二选一由实现定，但必须可渲染）：严格按 7.3.3，顺序列出 `起点→首景点→…` 的每一段，每元素含 `from`、`to`、`distance_km`、`duration_min`、`recommended_mode`、`cost_estimate_cny`、`data_quality`（`routed` / `estimated`）、`notes``weather_hint`（可选）：当日天气一句摘要 + 穿搭/携带雨具建议7.5.3 汇总块`pois_summary[]`、`legs` 的聚合视图（可选 `legs_flat[]` 便于导出）、`transit_matrix`（与 `legs` 勿重复矛盾，矩阵可为辅助）、`lodging_zones[]`、`tips`（应急电话等）天气与动线一致性声明：若用户修改日期后未重新生成，须提示「天气与行程可能未同步更新」---7.6 地图可视化（MVP1）标记：当日起点、景点、餐饮、住宿区域中心。连线：严格按 `legs[]` 顺序绘制折线，与文字「距离、用时」一致；按日分色。交互：点击节点或边查看 Leg 详情卡片；外链导航（起点→终点预填）。---7.7 多方案对比（MVP1）默认 2～3 套；并列展示：总览、预算区间、动线特点、住宿区域差异、每日天气摘要 与 因天气产生的选点差异（对照 7.3.5）。至少一套可定位为 「天气保守型」（优先室内与短户外），另一套可为 「经典晴好型」（在预报为晴时与保守型收敛，系统可合并为 2 套并说明）。切换不丢失已填问卷（同会话）；切换方案时 Leg 与地图联动刷新。---7.8 手动编辑（MVP2）增删改景点、拖拽排序 → 触发重算（全量或增量由技术设计）；`legs[]` 与地图同步更新。替换住宿区域或 修改当日起点 → 重算当日全部 Leg，并更新通勤摘要。操作撤销栈（建议）：至少支持上一步撤销。---7.9 实时数据（MVP2）天气：在 MVP1「按生成时刻预报驱动排程」基础上，增加 行前刷新、极端天气预警、可选 一键重排（重新跑 7.3.5 + Leg 重算）；穿搭与携带品建议随刷新更新。景点状态：依赖数据源；无数据则隐藏模块。公交实时：默认不做；若做需单独数据源与性能预算。---8. 非功能需求类别	要求性能	MVP0：问卷提交后 90 秒内 返回首屏可阅读结果（P95，不含用户网络极差场景）；可异步加载地图。可用性	问卷支持保存草稿（有账号或本地）；移动端关键路径可用。安全	输入过滤、输出 XSS 防护；API Key 服务端保管。隐私	最小化收集；隐私政策说明位置与行程数据的用途与保留期。可观测	生成失败率、平均耗时、问卷放弃率、方案切换率、天气 API 成功率、Leg 估算 vs 路网占比。无障碍	关键按钮与对比度遵循 WCAG 基础建议（随 Web 版本推进）。---9. AI 与工作流（Coze / 智能体）9.1 分工建议LLM：自然语言润色、避坑文案、根据槽位生成可读攻略；不单独承担硬约束排程。规则引擎 / 代码节点：开放时间窗校验、互斥校验、每日负载、距离矩阵调用；天气归一与 POI 权重调整（7.3.5）。优化模块：启发式排序 + 可行性修复循环；目标函数含 天气惩罚项（可配置权重）。9.2 Prompt 边界必须传入 已校验的 JSON 槽位 + 已确定的 POI 列表与时间轴草稿 + （若有）每日天气结构；LLM 主要负责叙述与风险提示，避免「幻觉景点」。9.3 机器可读配置（附录 A 概要）问卷定义：`questions[]` 每项含 `id`、`type`、`options`、`depends_on`、`validation`。输出定义：`TravelPlan` JSON Schema（版本号
