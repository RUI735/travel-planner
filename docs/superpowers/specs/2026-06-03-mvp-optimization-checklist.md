# Travel Planner MVP 功能待优化清单

> 综合来源：原始 PRD `旅游规划1.0.txt` + 当前代码实现审查 + PRD `2026-05-16-travel-planner-prd.md`

---

## 🔴 P0 — 体验决定性缺口

### 1. 日期选择器 → 原生控件
- **现状**：纯文本输入 `YYYY-MM-DD`，易输错
- **原始 PRD**：7.1.2 基础槽位均使用明确控件
- **目标**：使用 React Native 日期滚轮或日历弹出组件

### 2. 出行人数输入
- **现状**：无此字段
- **原始 PRD**：7.1.2 基础槽位 — 数字步进器，默认 2 人
- **目标**：创建页加数字步进器，影响人均预算换算

### 3. 预算档位输入
- **现状**：无此字段
- **原始 PRD**：7.1.2 基础槽位 — 档位 + 自定义金额，默认经济型
- **目标**：档位选择器（经济/舒适/轻奢）+ AI 感知预算约束

### 4. 每日预算摘要（输出侧）
- **现状**：行程输出无预算信息
- **原始 PRD**：7.5.1 `budget_summary`（门票/交通/餐饮/住宿参考区间）
- **目标**：AI 在行程中输出每日大致花销区间

### 5. 天气穿搭/携带建议
- **现状**：WeatherBanner 仅展示温度和降水概率
- **原始 PRD**：7.5.2 `weather_hint` — 当日天气一句摘要 + 穿搭/携带雨具建议
- **目标**：WeatherBanner 加一行 AI 生成的穿搭/携带建议文字

### 6. 搜索添加景点（P1 → 提前到 P0）
- **现状**：Store 有 `addSpot`，UI 无搜索入口，仅有删除按钮
- **原始 PRD**：7.8 手动编辑 — 增删改景点
- **目标**：行程详情页加搜索框 → 高德 POI 搜索 → 选中添加 → 重算路线

## 🟡 P1 — 差异化竞争力

### 7. AI 生成多方案（2 套）
- **现状**：仅生成 1 套方案
- **原始 PRD**：7.7 多方案对比 — 默认 2~3 套，并列展示；至少 1 套天气保守型 + 1 套晴好型
- **目标**：生成 2 套方案 — 经典版 + 天气友好版，用户可切换查看

### 8. 天气驱动排程（可解释）
- **现状**：天气仅事后标注预警，不参与景点筛选/排序
- **原始 PRD**：7.3.5 天气驱动 — 阴天降 `coastal`/`scenic_viewpoint` 权重、升 `indoor`；可解释
- **目标**：AI System Prompt 中引入天气权重规则，天气参与选点，输出可解释理由

### 9. Leg 补全（当日起点 → 首景点、末景点 → 住宿）
- **现状**：仅相邻景点间有 RouteSegment，缺少首/末段
- **原始 PRD**：7.3.3 强制完整 Leg 链 — 起点→景点1→…→住宿
- **目标**：每天输出完整的逐段 Leg，含起点和终点

### 10. 拖拽排序景点
- **现状**：Store 有 `reorderSpots`，UI 无交互
- **原始 PRD**：7.8 手动编辑 — 拖拽排序 → 触发路线重算
- **目标**：SpotCard 列表支持长按拖拽排序

### 11. 景点删除撤销
- **现状**：删除即永久
- **原始 PRD**：7.8 操作撤销栈 — 至少支持上一步撤销
- **目标**：删除后底部 Toast "已移除，点击撤销"，3 秒后消失

### 12. 路线不合理标记优化
- **现状**：仅按单段 >30min 判断，逻辑简单
- **原始 PRD**：7.3.2 路线优化 — 在开放时间窗与每日强度上限内启发式排序
- **目标**：综合距离、方向一致性（往返检测）、时间窗判断路线合理性

## 🟢 P2 — 后续迭代

### 输入层深化

### 13. 槽位可跳过 + 默认推断
- **原始 PRD**：7.1.1/7.1.4 — 基础槽位均可跳过，走默认推断策略
- **目标**：每个字段加"跳过"选项；未填时用默认值并在输出中标注

### 14. 主人群选择
- **原始 PRD**：7.1.3 Step A — 单人/情侣/亲子/老人/朋友/家庭
- **目标**：单选 + 补充标签多选（老人同行/婴幼儿/轮椅辅助）

### 15. 旅行风格选择
- **原始 PRD**：7.1.3 Step A — 休闲舒适/特种兵/深度文化/省钱/打卡拍照等
- **目标**：风格决定每日景点数约束（休闲 2-3 / 特种兵 4-5）和节奏

### 16. 特殊限制选择
- **原始 PRD**：7.1.3 Step A — 不爬山/不早起/少走路/不坐公交等
- **目标**：多选，进入约束引擎（不爬山→过滤山地标签；不早起→首活动 ≥9:00）

### 17. 交通偏好选择
- **原始 PRD**：7.1.3 Step C — 省时/省钱/便捷公交/自驾
- **目标**：偏好像影响路线计算 mode 选择，目前固定驾车

### 18. 当日起点输入
- **原始 PRD**：7.1.2 — 地址搜索/地图选点/快捷标签；可跳过
- **目标**：每天可设不同起点（酒店/车站/机场），驱动首段 Leg

### 输出层深化

### 19. 住宿区域推荐 + 通勤摘要
- **原始 PRD**：7.4 — MVP0 输出住宿区域（商圈名称）+ 理由 + 与动线通勤摘要
- **目标**：AI 输出中增加 `lodging_zones` 块

### 20. 可解释块
- **原始 PRD**：7.5.1 — `optimization_goal`、`active_constraints[]`、`tradeoff_notes`
- **目标**：行程顶部展示"为什么这样排"的可折叠说明

### 21. 餐饮推荐嵌入 Leg
- **原始 PRD**：7.3.4 — 午/晚各 0-1 推荐位，作为独立 Leg 节点
- **目标**：AI 输出中嵌入餐饮推荐，参与动线序列

### 22. 每日缓冲时间
- **原始 PRD**：7.3.3 — 景点间插入交通缓冲与简餐/休息块
- **目标**：Spot 之间增加缓冲 gap，避免行程过于紧凑

### 23. 免责声明
- **原始 PRD**：7.5.1 — `risk_disclaimers`（数据时效、预约责任、天气仅供参考）
- **目标**：行程底部固定展示免责文字

### 24. 约束冲突提示
- **原始 PRD**：7.1.5 — 互斥拦截（如"不坐公交"与"公交为主"冲突）
- **目标**：提交前校验 + 冲突说明 + 放宽建议

### 25. 景点筛选可解释
- **原始 PRD**：7.3.1 — 候选集 + 剔除原因日志
- **目标**：可选展示"为什么排除了某些景点"
