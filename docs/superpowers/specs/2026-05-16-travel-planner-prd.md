# Travel Planner PRD

## 1. 概述

**产品背景：** 当前年轻旅行者做攻略主要依赖小红书、携程等平台手动搜集信息，或使用 AI 工具生成行程。但这些方案普遍缺乏实用性——不考虑实时天气变化、景点实际开放状态、携带证件等关键细节，也不计算景点间实际路程和交通耗时，导致"攻略好看但走不下来"。用户需要一款能真正"无脑跟走"的智能规划工具。

**产品目标：** 5 天内交付 MVP 版本，验证"细节驱动的 AI 行程规划"核心价值。

**产品范围：** 单用户、本地存储的 App，输入目的地+日期后 AI 自动生成考虑天气+地图+景点提醒的每日行程，支持编辑调整。

**目标用户：** 18-26 岁大学生/研究生/初级职场人，预算有限但精力充沛，喜欢打卡体验，出行前集中规划、途中可能每日打开调整。

**商业模式：** MVP 阶段免费。后续如需付费采用订阅制。

**技术方案：** App 形态（iOS/Android），免登录，本地存储，接入天气API、地图API、AI大模型。

## 2. 功能列表

| 功能模块 | 功能点描述 | 优先级 |
|---------|-----------|--------|
| 行程生成 | 输入目的地、旅行日期、偏好标签，AI 自动生成每日行程（景点+美食+交通路线） | P0 |
| 天气整合 | 接入天气 API，按日期展示天气；遇恶劣天气建议调整方案及理由，用户自主决定是否采纳 | P0 |
| 景点提醒 | 展示景点的闭馆日、开放时间、证件要求、预约提醒等注意事项 | P0 |
| 地图路线 | 每日行程内景点间在地图上连线，计算实际路程和交通时长，标记不合理路线 | P0 |
| 行程编辑 | 支持手动增删景点、调整顺序、修改备注，行程保存到本地 | P1 |
| 行程总览 | 以天为单位查看完整行程，显示每日概览（景点数、天气、总耗时） | P1 |

## 3. 功能详细说明

### 3.1 行程生成（P0）

**(1) 用户流程**

Happy Path:
1. 用户打开 App，进入首页
2. 输入目的地（城市名）、选择旅行日期范围、可选勾选偏好标签（美食/人文/自然/打卡拍照/悠闲）
3. 点击"生成行程"
4. App 调用天气 API 获取目的地对应日期的天气、调用 AI 大模型生成每日行程
5. 行程生成完毕，按"天"展示行程卡片列表
6. 用户点击某一天查看详细行程（景点列表+地图路线+天气+提醒）

Unhappy Paths:
- 用户未输入目的地就点击生成 → 提示"请输入目的地"
- 用户未选择日期就点击生成 → 提示"请选择旅行日期"
- 目的地名称模糊或不存在 → 提示"未找到该目的地，请重新输入"
- AI 生成超时（>30s）→ 提示"生成失败，请重试"，提供重试按钮
- 天气 API 请求失败 → 跳过天气相关逻辑，生成不含天气信息的行程，顶部提示"暂无法获取天气信息"
- AI 返回内容格式异常 → 提示"生成失败，请稍后重试"
- 网络完全断开 → 提示"网络连接失败，请检查网络后重试"

**(2) 状态机**

行程对象状态：

| 状态 | 进入条件 | 退出/转换 |
|------|---------|----------|
| 未生成 | 用户首次打开或清空行程 | 用户点击"生成行程"且参数齐全 → 进入"生成中" |
| 生成中 | 触发行程生成请求 | 成功 → "已生成"；超时/失败 → "生成失败" |
| 已生成 | AI 返回有效行程数据 | 用户手动编辑行程 → "已修改" |
| 已修改 | 用户对已生成行程进行增删改 | 持久化本地 |
| 生成失败 | 请求超时或返回异常 | 用户重试 → 回到"生成中"；用户放弃 → "未生成" |

天（Day）对象状态：
- 未展开：行程列表中的默认状态
- 已展开：用户点击某天进入详情

**(3) 字段规范**

| 字段名称 | 是否必填 | 数据类型 | 长度限制 | 校验规则 | 报错提示语 |
|---------|---------|---------|---------|---------|-----------|
| 目的地 | 是 | 文本 | 2-20字 | 不为空，仅中英文/数字 | "请输入目的地" / "目的地名称无效" |
| 开始日期 | 是 | 日期 | — | 不能早于今天 | "请选择出发日期" |
| 结束日期 | 是 | 日期 | — | 不能早于开始日期 | "结束日期不能早于出发日期" |
| 偏好标签 | 否 | 多选 | 0-3个 | 预置选项 | — |
| 每日景点数上限 | 否（默认4个） | 数字 | 2-8 | 整数 | — |

**(4) 文案规范**

文案人格：简洁实用型，像朋友一样直接但有温度，不做卖萌。

| 场景 | 文案 |
|------|------|
| 空状态（首次打开） | "还没有行程计划，输入目的地开始规划吧" |
| 加载中（生成中） | "正在规划行程，预计需要 10-20 秒..." |
| 成功（生成完成） | "行程已生成，点击查看详情" |
| 失败（生成失败） | "生成失败，请重试" |
| 网络断开 | "网络连接失败，请检查你的网络" |

**(5) 异常处理**

| 异常场景 | 系统处理 |
|---------|---------|
| 网络断开 | 即时检测，Toast 提示"网络连接失败，请检查你的网络"，禁用生成按钮 |
| API 超时（>30s） | 自动终止请求，展示失败状态+重试按钮 |
| 无数据返回 | 展示"该目的地暂不支持，试试其他城市" |
| 重复提交（快速点击） | 生成中时按钮置灰+loading动效，不允许重复点击 |
| 前端校验失败 | 对应字段下方红色提示文字 |
| AI 校验失败 | 展示 AI 返回的具体错误（如有），默认兜底"生成失败，请稍后重试" |

### 3.2 天气整合（P0）

**(1) 用户流程**

Happy Path:
1. 用户查看某天行程详情时，顶部区域展示当天天气：天气图标、温度范围、天气状态文字
2. 如当天有恶劣天气（大雨/暴雪/高温/台风等），该天行程卡片出现黄色/红色标记
3. 系统展示受影响景点 + 调整理由 + 建议替代景点
4. 用户可一键替换或忽略建议——系统不会自动修改用户行程

Unhappy Paths:
- 天气 API 请求失败 → 天气区域显示"暂无法获取天气"
- 目的地仅支持有限天数预报（如7天）→ 超出范围的日期显示"暂无预报"
- 用户离线查看已有行程 → 展示最后一次获取的天气数据 + "上次更新于 xx:xx"

**(2) 状态机**

天气数据状态：

| 状态 | 进入条件 | 退出/转换 |
|------|---------|----------|
| 加载中 | 进入行程详情页，天气API请求中 | 成功→"已获取"；失败→"获取失败" |
| 已获取 | API返回有效数据 | 超24h自动刷新 |
| 获取失败 | API超时/报错 | 用户下拉刷新→回到"加载中" |
| 数据过期（>24h） | 上次获取超过24小时 | 用户查看时自动触发刷新→"加载中" |

天气影响分级：

| 等级 | 条件 | UI表现 | 用户操作 |
|------|------|--------|---------|
| 无影响 | 天气正常 | 无特殊标记 | 无 |
| 轻度影响 | 小雨/降温/阴天 | 黄色角标 + 理由说明 + 建议替代 | 可替换/忽略 |
| 严重影响 | 大雨/暴雪/台风/高温 | 红色角标 + 横幅提示 + 理由说明 + 建议替代 | 可替换/忽略 |

**(3) 字段规范**

| 字段名称 | 数据来源 | 数据类型 | 备注 |
|---------|---------|---------|------|
| 日期 | 行程数据 | Date | 关联到该天天气 |
| 天气状态 | 天气API | 枚举（晴/多云/阴/小雨/中雨/大雨/暴雪/台风等） | 映射为图标 |
| 最高温度 | 天气API | 数字（℃） | |
| 最低温度 | 天气API | 数字（℃） | |
| 降水量 | 天气API | 数字（mm） | 可选展示 |
| 预警等级 | 天气API | 枚举（无/黄色/橙色/红色） | 决定UI标记颜色 |

**(4) 文案规范**

| 场景 | 文案 |
|------|------|
| 天气正常 | "{天气状态}，{高温}℃ / {低温}℃，适合出行" |
| 轻度影响+建议 | "今天{阴天}，{海边}景色可能打折扣，建议调整为{XX博物馆}" |
| 严重影响+建议 | "明天有{大雨预警}，建议调整为室内景点如{XX美术馆}" |
| 加载失败 | "暂无法获取天气" |
| 数据过期 | "天气信息上次更新于 xx:xx，下拉刷新" |
| 用户拒绝调整 | 无提示，保留原行程 |
| 用户接受调整 | Toast"已替换为 {替代景点}" |

**(5) 异常处理**

| 异常场景 | 系统处理 |
|---------|---------|
| 天气API超时 | 5秒超时，降级展示"暂无法获取天气"，不影响行程其他功能 |
| API密钥失效 | 静默失败，天气区域显示"暂无法获取天气" |
| 解析天气数据失败 | 降级处理，不阻塞页面 |
| 多天行程超出预报范围 | 超出天数显示"暂无预报"，不报错 |

### 3.3 景点提醒（P0）

**(1) 用户流程**

Happy Path:
1. 行程详情页中，每个景点卡片下方展示"注意事项"行
2. 提醒内容包括：开放时间、闭馆日、证件要求、预约方式
3. 如景点在某天闭馆，该景点卡片红色高亮，顶部标记"此日闭馆"
4. 如景点需携带证件（学生证/身份证），提醒行展示黄色图标

Unhappy Paths:
- AI 未返回某景点的提醒信息 → 仅展示基本信息，提醒区域空白，不做错误提示
- 景点信息可能不准确 → 标记"建议核实"

**(2) 状态机**

景点提醒状态：

| 状态 | 进入条件 | 退出/转换 |
|------|---------|----------|
| 正常 | 日期不是闭馆日，无特殊要求 | — |
| 需注意 | 有证件/预约要求 | 用户查看后无变化 |
| 闭馆冲突 | 日期命中闭馆日 | 用户手动替换该景点→回到"正常" |

**(3) 字段规范**

| 字段名称 | 是否必填 | 数据类型 | 来源 |
|---------|---------|---------|------|
| 景点名称 | 是 | 文本 | AI生成 |
| 开放时间 | 否 | 文本 | AI知识 |
| 闭馆日 | 否 | 文本（如"每周一"） | AI知识 |
| 证件要求 | 否 | 文本（如"学生证"） | AI知识 |
| 预约要求 | 否 | 文本（如"需提前1天微信公众号预约"） | AI知识 |
| 注意事项 | 否 | 文本 | AI知识 |

**(4) 文案规范**

| 场景 | 文案 |
|------|------|
| 有提醒信息 | 逐条列出，每条带对应图标（⏰ 🚫 🪪 📞） |
| 无提醒信息 | 不展示提醒区域 |
| 闭馆冲突 | "该景点在 {日期} 闭馆，建议替换或调整日期" |
| 建议核实 | "以下信息可能不准确，建议出发前核实" |

**(5) 异常处理**

| 异常场景 | 系统处理 |
|---------|---------|
| AI未返回该景点提醒 | 优雅降级，不展示提醒行，不报错 |
| 提醒数据格式异常 | 忽略异常字段，仅展示正常字段 |

### 3.4 地图路线（P0）

**(1) 用户流程**

Happy Path:
1. 用户进入每日行程详情页，上方缩略地图标注当日所有景点位置，按顺序连线
2. 每段路线标注预估交通时长（驾车/公共交通）和距离
3. 路线不合理（来回跑/绕路）时该段标红并提示"路线可优化"
4. 用户点击某段路线展开交通详情
5. 用户拖拽调整景点顺序，路线重新规划

Unhappy Paths:
- 地图API请求失败 → 仅展示景点列表，不展示地图
- 两景点间无法计算路线 → 显示"无法计算路线"
- GPS关闭 → 不影响查看行程

**(2) 状态机**

| 状态 | 进入条件 | 退出/转换 |
|------|---------|----------|
| 加载中 | 地图SDK+路线计算中 | 成功→"已加载"；失败→"加载失败" |
| 已加载 | 地图+路线渲染完毕 | 用户调整顺序→"加载中" |
| 加载失败 | API超时 | 降级为纯列表模式 |
| 路线合理 | 路线顺序合理 | — |
| 路线不合理 | 路径存在往返/绕路 | 用户调整顺序→"已加载" |

**(3) 字段规范**

| 字段名称 | 是否必填 | 数据类型 | 来源 |
|---------|---------|---------|------|
| 景点经纬度 | 是 | float×2 | 地图API Geocoding |
| 景点间距离 | 否 | 数字（km） | 地图API |
| 预估驾车时间 | 否 | 数字（分钟） | 地图API |
| 预估公交时间 | 否 | 数字（分钟） | 地图API |
| 路线顺序 | 是 | int | AI生成，用户可调整 |

**(4) 文案规范**

| 场景 | 文案 |
|------|------|
| 正常路线 | "🚗 约 {x} 分钟  🚌 约 {y} 分钟" |
| 路线可优化 | "当前路线绕路，建议调整为以下顺序..." |
| 无法计算 | "两地间无法计算交通路线" |
| 步行可达 | "步行可达，约 {x} 分钟" |
| 地图加载失败 | "地图暂不可用" |

**(5) 异常处理**

| 异常场景 | 系统处理 |
|---------|---------|
| 地图SDK加载失败 | 降级展示纯景点列表 |
| 某个景点Geocoding失败 | 灰色点标注，不参与路线连线 |
| 两景点距离过近（<500m） | 标注为"步行可达" |

### 3.5 行程编辑（P1）

**(1) 用户流程**

Happy Path:
1. 用户在行程详情页对景点进行增删/替换/调整顺序/添加备注
2. 每次变更自动保存到本地，无需手动点保存
3. 点击"+ 添加景点"，搜索框输入景点名 → 调用地图API搜索 → 选中添加
4. 左滑景点卡片出现"删除"按钮

Unhappy Paths:
- 搜索景点无结果 → 提示"未找到该景点，换个关键词试试"
- 删除最后一个景点 → 该天展示空状态
- 存储空间不足 → 提示"存储空间不足，请清理后重试"

**(2) 状态机**

| 状态 | 进入条件 | 退出/转换 |
|------|---------|----------|
| 原始（AI生成） | 行程首次生成 | 用户编辑→"已修改" |
| 已修改 | 用户做了任何增删改 | 持久化到本地 |
| 保存中 | 本地写入进行中 | 成功→"已修改"；失败→"保存失败" |
| 保存失败 | 写入异常 | 重试→"保存中" |

**(3) 字段规范**

| 字段名称 | 是否必填 | 数据类型 | 校验规则 |
|---------|---------|---------|---------|
| 景点名称 | 是（添加时） | 文本 | 不为空 |
| 备注 | 否 | 文本 | 最多200字 |
| 交通偏好 | 否 | 枚举（驾车/公交/步行） | — |

**(4) 文案规范**

| 场景 | 文案 |
|------|------|
| 空天 | "这天还没有安排，点击下方添加景点" |
| 保存失败 | "保存失败，请重试" |
| 删除 | 左滑直接删除+底部"已移除，点击撤销" 3秒后消失 |
| 搜索无结果 | "未找到该景点，换个关键词试试" |

**(5) 异常处理**

| 异常场景 | 系统处理 |
|---------|---------|
| 本地存储满 | Toast"存储空间不足" |
| 快速连续删除添加 | 每次操作独立，撤销仅针对最后一次操作 |
| App被系统杀死 | 每次变更立即写入本地 |

### 3.6 行程总览（P1）

**(1) 用户流程**

Happy Path:
1. 首页即为行程总览页，按天展示行程卡片列表
2. 每天卡片展示：日期、星期、天气图标+温度、景点个数、预估总耗时
3. 顶部展示：目的地、日期范围、总天数
4. 点击某天卡片进入详情
5. 长按可拖拽交换天行程

Unhappy Paths:
- 无行程 → 空白首页，引导"输入目的地开始规划"

**(2) 状态机**

| 状态 | 进入条件 | 退出/转换 |
|------|---------|----------|
| 空 | 无任何行程数据 | 生成行程→"有行程" |
| 有行程 | 本地存在有效行程 | 用户清空→"空" |

**(3) 文案规范**

| 场景 | 文案 |
|------|------|
| 空状态 | "还没有行程计划" + CTA按钮"开始规划" |
| 有行程 | 顶部："{目的地} · {n}天{n-1}晚" |
